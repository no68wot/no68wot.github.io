Oracle VirtualBox - How to remove ?

Auflisten aller VirtualBox-Pakete: 
dpkg -l | grep virtualbox
Entfernen aller VirtualBox-Pakete (die mit virtualbox- beginnen): 
sudo apt-get remove virtualbox-\*
oder: 
sudo apt-get purge virtualbox-\*
oder (komplett): 
sudo apt-get remove virtualbox*
sudo apt-get purge virtualbox*
wobei virtualbox-\* bedeutet alle Matches, die den Term virtualbox enthalten. Das Backslash (\) bewirkt das Escape von Wilcards. Ausführliche Erklärung für Backslash (\) mit Stern (*) findet sich unter: https://askubuntu.com/questions/190004/how-to-uninstall-virtualbox-in-12-04/190053#190053

Oracle VirtualBox - How to install (Ubuntu 16.04 Xenial) ?

Mit uname werden Systeminformationen (Kernel, Prozessor, Netzwerkname etc.) ausgegeben.

Linux-Kernel herausfinden: 
uname -r
4.4.0-78-generic
32bit- oder 64bit-Betriebssystem: 
uname -p
x86_64
oder: 
uname -m
x86_64
oder: 
uname -i
x86_64
→ Das ist ein 64bit-Betriebssystem, wegen "_64".

Distribution-Version ermitteln: 
cat /etc/lsb-release
DISTRIB_ID=Ubuntu
DISTRIB_RELEASE=16.04
DISTRIB_CODENAME=xenial
DISTRIB_DESCRIPTION="Ubuntu 16.04.2 LTS"
→ Linux-Version ist Ubuntu 16.04 (Xenial)

Ubuntu ist eine Debian-basierte Linux-Distribution.

Öffne die Datei /etc/apt/sources.list : 
sudo gedit /etc/apt/sources.list
Füge die folgende Zeile in die Datei ein: 
deb http://download.virtualbox.org/virtualbox/debian xenial contrib
Für andere Distributionen, ersetze xenial durch yakkety, vivid, utopic, trusty, raring, quantal, precise, lucid, jessie, wheezy oder squeeze

Oracle-Public-Key für apt-secure herunterladen:
- Ubuntu 16.04 (Xenial) und neuer: oracle_vbox_2016.asc

Diesen Schlüssel (Public-Key) registrieren: 
sudo apt-key add oracle_vbox_2016.asc
oder kombiniere Herunterladen und Registrieren des Public-Keys: 
wget -q https://www.virtualbox.org/download/oracle_vbox_2016.asc -O- | sudo apt-key add -
Fingerprint für oracle_vbox_2016.asc ist:

Oracle Corporation (VirtualBox archive signing key)
System zuerst updaten: 
sudo apt-get update
Installiere die dkms-Pakete für Virtual-Host-Kernel-Module (vboxdrv, vboxnetflt, vboxnetadp): 
sudo apt-get install dkms
System noch mal updaten: 
sudo apt-get update
VirtualBox 5.1 installieren: 
sudo apt-get update

Fehlerbehandlung : Wenn es Problem bei der Installation oder beim Starten von VirtualBox gibt, z.B. Kernel driver not installed (rc=-1908)

kann man es durch eine der folgenden Methoden beheben:
  1. Als root ausführen: sudo /sbin/vboxconfig
  2. Falls Methode I nichts bringt, liegt das Problem höchstwahrscheinlich an UEFI-Secure-Boot. Mit dem Befehl lsmod | grep vboxdrv kann überprüft werden, ob die VirtualBox-Module (vboxdrv, vboxnetflt, vboxnetadp) erfolgreich geladen werden. (Befehl lsmod listet Informationen über Modulen in Linux- Kernel auf). Falls durch lsmod | grep vboxdrv nichts auf der Konsole angezeigt werden, liegt es nahe, dass die o.g. Module wegen aktivem Secure-Boot nicht geladen werden können und einen Start von VirtualBox hindert.
    1. Entweder deaktiviert man das Secure-Boot, indem man im BIOS die Option UEFI- Secure-Boot auf Disable setzt. Danach soll VirtualBox problemlos laufen können.
    2. Oder man signiert die notwendigen Module (vboxdrv, vboxnetflt, vboxnetadp) jeweils mit dem X.509 Schlüssel - ohne EFI-Secure-Boot zu deaktivieren.

      Man führt /sbin/vboxconfig als Root aus, um den Setup-Prozess zu wiederholen: 
      sudo /sbin/vboxconfig
        vboxdrv.sh: Stopping VirtualBox services.
        vboxdrv.sh: Building VirtualBox kernel modules.
        vboxdrv.sh: Starting VirtualBox services.
        vboxdrv.sh: Building VirtualBox kernel modules.
        vboxdrv.sh: failed: modprobe vboxdrv failed. Please use 'dmesg' to find out why.
      
        There were problems setting up VirtualBox.  To re-start the set-up process, run 
        /sbin/vboxconfig as root.
      In der vorletzten Zeile der Ausgabe "vboxdrv.sh: failed: modprobe vboxdrv failed. Please use 'dmesg' to find out why" wird auf 'dmesg' bzw. 'modprobe vboxdrv' hingedeutet.

      Mit dmesg kann man Kernel-Ring-Puffer anzeigen lassen oder kontrollieren. Aber jede Ausführung von dmesg liefert viele lange Ausgaben, die teilweise unübersichtlich sind und der Lösung nicht viel beiträgt. Stattdessen wird der Befehl modprobe vboxdrv in Betracht gezogen. Der Befehl modprobe wird verwendet, um ein lädbares Kernel-Modul (LKM, loadable kernel module) in Linux-Kernel hinzufügen oder ein LKM aus dem Linux-Kernel zu entfernen.

      Ausführung von modprobe vboxdrv als Root liefert: 
      sudo modprobe -v vboxdrv
        insmod /lib/modules/4.4.0-78-generic/misc/vboxdrv.ko 
        modprobe: ERROR: could not insert 'vboxdrv': Required key not available
      Die Ausgabe Required key not available bedeutet, dass der notwendige Schl¨ssel (hier X.509) fehlt, das Modul 'vboxdrv' nicht signiert ist und deswegen nicht geladen werden kann [4]. Die Lösung besteht einen X.509-Schlüssel zu erzeugen und damit das Modul 'vboxdrv' zu signieren [1][4].

      Erzeugen des Private/Public-Schlüsselpaars (X.509) mit OpenSSL: 
      openssl [command] [command-options] [command-arguments]
          req : PKCS#10 X.509 Certificate Signing Request (CSR)	Management
                PKCS = Public Key Cryptography Standard
                Jedes Mal wenn ein neues Zertifikat beantragt (benötigt) wird, wird auch ein neuer CSR 
                erstellt. Der CSR ist quasi ein Antrag für ein neues Zertifikat. Für die Erstellung
                von CSR wird ein privater Schlüssel (Private Key) benötigt. Jeder neuer
                CSR muss von einer CA (Certificate Authority) oder Selft-Signed signiert werden [5].
          req -new   : Diese Option erstellt ein neues CSR, also einen Antrag für ein neues Zertifikat. 
          req -nodes : Diese Option bedeutet nicht den englischen Knoten (node), sondern eher no DES,
                       also kein DES (Data Ecryption Standard). Bei der Angabe der Option -nodes wird
                       OpenSSL den Private-Key nicht kodieren unter Format PKCS#12 (.p12, .pfx, X.509) [7].
          openssl req -new -x509 -newkey rsa:2048 -keyout MOK.priv -outform DER -out MOK.der -nodes -days 36500 -subj "/CN=Descriptive name/"

Formatter Bibliography

For a long time now, I wish an easy way to get my links (in plaintext form) automatically formatted as hyperlinks (i.e. blue-colored and unterlined).
For example, if I just write the hyperlink www.myHyperlink.com, it should be automatically formatted as www.myHyperlink.com.
Furthermore, if there are more than one hyperlinks, e.g.

www.myHyperlink01.com
www.myHyperlink02.com
www.myHyperlink03.com

they should be formatted and numbered consecutively, like

[1] www.myHyperlink01.com
[2] www.myHyperlink02.com
[3] www.myHyperlink03.com

For this purpose, I use a HTML-<table> with separated rows (<tr>) for the hyperlinks. Each row has only one cell/column (<td>). Each hyperlink is inserted into one cell. The table gets a class-attribute with value "tabBib". 

<table class="tabBib">
 <tr>
   <td><a href="www.myHyperlink01.com">www.myHyperlink01.com</a></td>
 </tr>
 <tr>
   <td><a href="www.myHyperlink02.com">www.myHyperlink02.com</a></td>
 </tr>
 <tr>
   <td><a href="www.myHyperlink03.com">www.myHyperlink03.com</a></td>
 </tr>
</table>

Java : Object Reference und Typecasting

Gegeben seien folgende 3 Klassen (A.java, B.java, Main.java):

/* A.java */
public class A
{
  public void print(A a) {
    System.out.println("print(a) in A");
  }
}

/* B.java */
public class B extends A
{
  public void print(B b) {
    System.out.println("print(b) in B");
  }

  @Override
  public void print(A a) {
    System.out.println("print(a) in B");
  }
}

/**
 * Main.java 
 */
public class Main
{
  public static void main(String[] args) {
    B b = new B();
    A a = b;
    a.print(b); // print(a) in B
    a.print(a); // print(a) in B
    b.print(a); // print(a) in B
    b.print(b); // print(b) in B
  }
}

B ist Unterklasse (Subclass) von A, wegen B extends A. Deshalb erbt B alle Eigenschaften und Methoden von A und kann diese sogar überschreiben. Original-Methode in A und überschriebene/geerbte Methode in B werden im obigen Beispiel rot markiert.

In Main.java, betrachte zunächst nur die Anweisung new B(). Diese bewirkt, dass eine Instanz (Objekt) :B der Klasse B erzeugt wird, wie es die folgende Abbildung illustriert. 

                                new B();

  Klasse                         Objekt
+---------+    instance of     +---------+ 
|    B    |<-------------------|   :B    |
+---------+                    +---------+
Anstelle von instance of kann der UML-Stereotyp <<instance>> verwendet werden.

Dann wird durch die Zuweisung B b = new B() das erzeugte Objekt :B der Variable b zugewiesen. Die Variable b hat den Typ B und ist ein Pointer (Zeiger, Referenz) auf Objekt :B. 
                                           B b = new B();
  Klasse                         Objekt
+---------+    instance of     +---------+                 
|    B    |<-------------------|   :B    |<-------------- b
+---------+                    +---------+
Mit der Anweisung A a = b wird die Referenz von b der Variable a zugewiesen. Die Referenz von b ist das Objekt :B. Die Variable a ist ein Pointer vom Typ A und zeigt nun durch die o.g. Zuweisung ebenfalls auf Objekt :B. Also a zeigt auf das, was b auch zeigt. Die folgende Abbildung illustriert diesen Sachverhalt. Außerdem kann i.d.R. an dem erzeugten Objekt :B nichts mehr geändert werden. 
                                           B b = new B();
                                           A a = b = new B()
  Klasse                         Objekt
+---------+    instance of     +---------+                 
|    B    |<-------------------|   :B    |<-------------- b
+---------+                    +----^----+
                                    |
                                    +-------------------- a
Die Anweisung a.print(b) bewirkt, dass a (Typ A) zunächst scheinbar die Methode print(A a) der Klasse A aufrufen "soll". Diese Methode print(A a) von A wird bereits in der Klasse B überschrieben (@Override). Außerdem zeigt a (trotz Typs A) nur auf Objekt :B (Typ B). Deshalb kann a nicht auf die Original-Methode print(A a) in A zugreifen, sondern muss die geerbte und überschriebene Methode print(A a) in B aufrufen. (Die andere Methode print(B b) kommt für a nicht infrage, denn sie ist eigene Methode von B, und nicht von A). 
    Klasse
+------------+
|     A      |
+------------+
| print(A a) |
+------------+
      ^
     /_\
      |
      | extends                             B b = new B();
      |                                     A a = b = new B()
      |                            Objekt
+------------+    instance of     +---------+                 
|     B      |<-------------------|   :B    |<-------------- b
+------------+                    +----^----+
| print(A a) |                         |
| print(B b) |                         +-------------------- a
+------------+
Zusammengefasst: a ist vom Typ A, a muss Methode print(A a) von A aufrufen, kann aber nicht. Denn a ist kein Pointer (Referenz) auf irgendein Objekt :A der Klasse A. (Es gibt auch kein Objekt :A). Stattdessen ist a ein Pointer auf Objekt :B der Klasse B. Und die Methode print(A a) von A wird von B geerbt (B extends A) und überschrieben (@Override). Die beehrte Methode print(A a) existiert aslo auch im Objekt :B. So muss a (trotz Typs A) auf diese Methode print(A a) in B zugreifen.
Die Methode print(A a) erwartet ein Argument (Parameter) vom Typ A. In der Anweisung a.print(b) ist b (Typ B) das Argument. Wegen B extends A ist b auch vom Typ A. Und das genügt der Funktion print(A a) in B. Daher liefert die Anweisung a.print(b) die Meldung print(a) in B. 
a.print(b) ==> a.print(b : B extends A) ==> a.print(A a) ==> "print(a) in B"
                                              (in B)
Die nächste Anweisung a.print(a) verhält sich (fast) identisch wie die vorherige a.print(b). Der einzige Unterschied hier ist das Argument a (anstatt b). Da a vom Typ A ist, ergibt der Aufruf a.print(a) dieselbe Meldung print(a) in B. 
a.print(a) ==> a.print(a : A) ==> a.print(A a) ==> "print(a) in B"
                                    (in B)
Die nächste Anweisung b.print(a) kann wie folgt verstanden werden: b ist ein Pointer auf das (durch new B()) erzeugte Objekt :B. Mit b.print(..) wird eine gewisse Methode print(..) von B (bzw. :B) aufgerufen. Es gibt aber zwei solche (print(A a) und print(B b)) in B. Welche von denen gewählt wird, hängt vom Typ des erwarteten Arguments ab. In der Anweisung b.print(a) ist das Argument a vom Typ A. Also ist das jetzt klar, welche print-Methode aufgerufen werden soll, nämlich print(A a). 
b.print(a) ==> b.print(a : A) ==> b.print(A a) ==> "print(a) in B"
                                    (in B)
Die letzte Anweisung b.print(b) hat b (Typ B) im Argument. (Argument ist das was zwischen den Klammern steht). So wird die Methode print(B b) von B (bzw. :B) aufgerufen. 
b.print(b) ==> b.print(b : B) ==> b.print(B b) ==> "print(b) in B"
                                    (in B)
